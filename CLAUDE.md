# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

**Alphaclara** (package name `bullsignalsai`) is an Expo/React Native mobile app (iOS-first — tablet support is disabled, see `app.config.js`). It is the frontend-only half of a larger system: all financial data, AI signals, and pattern computation are produced by a separate FastAPI backend (`bullbrain-api`, hosted on Render) and Firestore. That backend's code does not live in this repo.

## Commands

```bash
npm run start      # expo start — Metro dev server
npm run ios         # expo run:ios
npm run android      # expo run:android
npm run web          # expo start --web
```

There is no configured lint, typecheck, or test script in `package.json` — do not assume `npm test`/`npm run lint` exist; check with the user before adding one. `/ios` and `/android` native folders are gitignored (generated by Expo prebuild), as is `.expo/` and `dist/`.

## Architecture

### Non-negotiable data flow rule

The UI must never call third-party financial/AI APIs (Finnhub, Grok, FMP, NewsAPI, OpenAI) directly, and must never perform heavy computation (pattern scanning, technical indicators) client-side. All of that happens ahead-of-time in backend cron jobs, written to Firestore, and the app only reads cheap, precomputed endpoints. Concretely:

```
UI (React Native) → services/*.js → FastAPI backend (API_BASE_URL) → Firestore (read-only)
```

`config/apiKeys.js` intentionally holds only `API_BASE_URL` (`https://bullbrain-api.onrender.com`); all other key constants are empty strings by design — "keys handled by backend" is a hard constraint, not an oversight. Do not add frontend API keys here. See `docs/stock-detail-architecture.md` for the full Firestore schema (`stock_realtime`, `stock_ai_snapshot`, `stock_smart_pattern`, `stock_technical_snapshot`, `stock_grok_summary`, `stock_news`) and per-section TTLs — useful when a screen needs to know how fresh a given field is expected to be.

### Main tab screens: data flow

Every main-tab screen follows the same shape: its `services/*.js` module fetches a snapshot endpoint, then does a second `GET /quotes-bulk?scope=<screen>&symbols=...` call for live prices on whatever symbols the first call returned, merges the two, and returns a defensively-normalized object (never throws — falls back to an empty-but-well-shaped result). The screen just renders it and polls on a fixed interval while focused; none of these use websockets or push-driven refresh.

| Screen | Service | Snapshot endpoint(s) | Poll interval |
|---|---|---|---|
| Home | `services/HomeService.js` | `GET /homescreen-data` (→ `getHomeScreen`), `GET /market-movers?mode=home` (→ `getHomeMovers`), `GET /verified-alpha` (→ `getVerifiedAlpha`) | 5s |
| Assets / Watchlist | `services/watchlistService.js` | `GET /watchlist/{userId}` (→ `getWatchlistScreen`) | 15s (aligned to backend quote-snapshot TTL) |
| Momentum | `services/MomentumService.js` | `GET /market-momentum` (→ `getMarketMomentum`); `POST /market-momentum/refresh` for manual/pull-to-refresh | 30s |
| Market | `services/MarketPulseService.js` | `GET /homescreen-context` (→ `getMarketOverview`), `GET /market-movers?mode=preview` + `GET /market-news` via `Promise.allSettled` | 30s |

Notes:
- **Bulk-quotes pattern**: always snapshot-fetch → collect symbols → one shared `quotes-bulk` call per screen (never N per-symbol requests). The `scope` param lets the backend apply different cache/priority rules per screen. Follow this pattern for any new screen that shows live prices.
- **`needs_refresh` / `session` flags**: quotes carry a `needs_refresh` boolean from the backend; services turn that into `session: "LIVE" | "LAST" | "PENDING"` so the UI can show staleness instead of pretending everything is real-time.
- Watchlist's `mergeWatchlistQuotes` additionally detects when a cached text summary ("under pressure") contradicts the live price move and regenerates the sentence rather than showing stale copy — a pattern worth reusing if other screens show AI-generated copy next to live prices.
- `services/MarketStatsService.js` (`fetchLiveMarketStats` → `/stats/live`) and `services/MarketFeedService.js` (`fetchMarketOverview` → `/market-overview`) are older, partially-used siblings of `MarketPulseService.js` — unlike the pattern above, they return hardcoded stub data on failure instead of an empty result. Prefer `MarketPulseService.js` for new Market-screen work.

### Layers

- **`screens/`** — one file per screen, wired together in `App.js` via a single `Stack.Navigator` (auth/onboarding flow) wrapping a `Tab.Navigator` (`MainTabs`: Home, Assets/Watchlist, Momentum, Market, Profile). Initial route is decided at launch by checking `AsyncStorage` for `userToken` / `onboarded` (see the `checkUserStatus` effect in `App.js`).
- **`services/`** — the data-access layer. Most files fetch from `API_BASE_URL` (the backend) and reshape the response for a specific screen (e.g. `HomeService.js` → HomeScreen, `stockDetailService.js`/`technicalDetailService.js`/`patternDetailService.js`/`decisionDetailService.js` → StockDetailScreen and its "Full Detail" sub-screens). Others wrap Firestore directly via `firebaseConfig.js` (auth, portfolio CRUD, cached summaries) or wrap push notifications (`pushNotificationService.js`).
- **`firebaseConfig.js`** — not just config: it initializes Firebase app/auth/Firestore *and* exports the Firestore data-access functions for portfolio positions (lot-based, FIFO) and cached AI summaries. Portfolio docs support a migration path from an old `{shares, avgCost}` schema to the current `{lots: [...]}` schema — `normalizeLots()` handles both transparently, so don't assume all stored docs are already lot-based.
- **`components/`** — shared UI (buttons, headers, chat/animated icon for "Astra" the AI assistant, market ticker, sparkline chart, toast).
- **`constants/theme.js`** and **`constants/typography.js`** — the canonical design tokens (`BRAND`, `TYPO`), used throughout `screens/` and `components/`. `styles/theme.js` is an older/divergent copy with fewer tokens (no `card2`, `softBorder`, `muted`, `green`, `blue`) — prefer `constants/theme.js` for new work and treat `styles/theme.js` as legacy unless told otherwise.
- **`utils/`** — formatting (`formatters.js`), signal/label helpers (`signalUtils.js`), and generated technical narrative text (`technicalNarrative.js`).
- **`hooks/useLiveQuotes.js`** — polling hook for realtime-ish price data.

### Client-side caching

`services/CacheManager.js` provides a generic `AsyncStorage`-backed TTL cache (`saveCache(key, data, ttlSeconds)` / `getCache(key)`) used to avoid re-hitting the backend more often than each section's documented TTL requires. `services/firestoreCache.js` is a Firestore-specific counterpart. When adding a new service call, check whether it should go through one of these rather than fetching on every render.

### Navigation conventions

New detail/sub-screens are added as `Stack.Screen` entries in `App.js` outside `MainTabs`, typically with a custom `headerBackImage` (an `Ionicons` chevron) and `BRAND.bg`/`BRAND.text` header styling — copy an existing entry (e.g. `FullTechnicalDetailScreen`) rather than inventing new header styling.

### Push notifications

`pushNotificationService.js` registers the device token against the backend keyed by Firebase `uid` (registered once, in `App.js`, after `initialRoute === "Main"`). Notification taps are routed by `type`/`symbol` payload fields in `handleNotificationNavigation` (`App.js`) — e.g. `watchlist_*`/`portfolio_position_big_move` types deep-link to `StockDetailScreen`, `portfolio_daily_performance`/`portfolio_ai_rebalance` go to the Portfolio tab, `crypto_market_move` goes to the Market tab. When adding a new notification type on the backend, add a matching branch here.
